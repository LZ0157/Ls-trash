# Track A Baseline 评测报告

> 用 NLI 判别模型替代原作者的 prompt + LLM 端到端方案，CPU only 跑 CDConv。
> 这是 Track A 最简形态的 MVP，用于在重设计方案之上拿到第一份对比数字。

## 一、配置

| 项 | 值 |
|---|---|
| 数据集 | `cdconv.txt` (CDConv，11660 条 JSONL) |
| 抽样 | 分层 50/类 = 200 条 (固定 seed=42) |
| NLI 模型 | `IDEA-CCNL/Erlangshen-Roberta-330M-NLI` (BERT-large, 24 层, 1024 hidden) |
| 推理设备 | **CPU only** (`CUDA_VISIBLE_DEVICES=""` 强制) |
| 句子切分 | 中英文标点 + 逗号细切 (`fine_split=True`) |
| 决策阈值 | contradiction prob ≥ 0.5 |
| 候选句对 | b2 内部 (intra) + b1↔b2 (cross) |
| 总 NLI 推理对数 | 1090 |
| **总耗时** | **21 秒** (≈ 9.6 样本/秒, 51 NLI 对/秒) |

## 三、结构性局限

### 1. Class 2 (角色混淆) 永远是 0
CDConv 的 `cdconv.txt` **不包含 persona 字段**，类 2 的判定需要"违背角色设定"信号，本基线没法获得。设计上把它合并到类 3 才公平。看"3 类合并"的指标更有意义。

### 2. Class 1 (句内矛盾) 召回率极低 (0.12)
CDConv 标的"句内矛盾"在我们的句子分割粒度下，**很多其实不在 b2 句子内部**，而是 b2 与 b1 之间的矛盾（CDConv 标注本身有噪声）。比如：

```
b1: 我已经大学毕业了，你呢？
b2: 是啊，可是我已经毕业了。      <- 标为 1，实际无矛盾
```

```
b1: 2只,其中一只是公的,另一只是母的
b2: 两只,一公一母,还有一只母的    <- 真正的句内矛盾 (2只 vs 3只)
```

第二例如果按逗号切，能切出 `[两只, 一公一母, 还有一只母的]` 三句，但 NLI 模型仍然没识别出"两"与"三只猫"的数量冲突——这是 NLI 模型在数字推理上的已知弱点。

## 四、Bad Case 分析

### A. False Positive (误报，30/50 类 0 被错判为有矛盾)

**主因：NLI 模型把"话题不相关"误判为"矛盾"**

```
b1: 看书           |  b2: 看电影           → p_contradict = 1.000  ❌
```
两个不同的爱好被判矛盾。

```
b1: 你是哪个学校,大几了?  |  b2: 我回家了      → p_contradict = 1.000  ❌
```
答非所问 ≠ 矛盾。

```
b1: 最喜欢罗纳尔多   |  b2: 我只知道我喜欢梅西  → p_contradict = 1.000  ❌
```
"只知道我喜欢梅西"被模型解读为排他性，实际并不真排斥同时喜欢罗纳尔多。

```
b1: 迪士尼是个很好的购物场所  |  b2: 不知道,是什么?  → p_contradict = 0.999  ❌
```
"不知道"被当作对存在性的否定。

**这一类误报占据了所有 FP 的绝大部分**——根因是 NLI 模型在"局部词义对立"上过于敏感，缺乏话题/语用层的理解。

### B. False Negative (漏报)

```
b1: 还好，不是很紧张。你呢？      
b2: 还好，只是不知道为什么会紧张    <- 标为 1，n_pairs=7 但全部判为无矛盾
```
"还好" 与"为什么会紧张"之间的反讽 NLI 模型抓不住。需要语用推理。

```
b1: 我要买电脑啊,想问问你的意见嘛,我对电脑不怎么了解
b2: 那你买电脑的时候是去专卖店买还是网店?       <- 标为 2 (角色混淆)
```
"我对电脑不怎么了解" 但 b2 反过来给用户出建议——这是**角色反转**而非语义矛盾，需要 persona 模型。

## 五、相比原方案的优势

> 注：原 prompt-based 方案的具体数字未保留（原 `eval_results_details*.json` 已删），无法做严格的同测试集对比。但从原作者 `analyze_results.py` 里写的分析逻辑可知，原方案存在严重的"模型偷懒全判 0"问题，2/3 类几乎为 0。

| 维度 | 原方案 (prompt + glm-4-flash) | Track A (NLI baseline) |
|---|---|---|
| 推理位置 | 调用付费 API | 全本地 CPU |
| 调用方式 | 一次 LLM 端到端分类 | 句子分割 + 多次 NLI 判别 |
| 速度 | ~0.3 sample/s (含 API 延时) | **9.6 sample/s** (CPU only) |
| 可解释性 | 一段自然语言 reason | **每条预测附带矛盾证据对** (前提-假设-置信度) |
| 模块化 | 单 prompt，整体调 | claim 分割 / NLI 模型 / 阈值 / 后处理规则 **每一层都可独立替换** |
| 失败模式 | 全判 0 (LLM 偷懒) | 过度激进（话题切换 → 误报） |
| 调参成本 | 改 prompt 反复试 | 改阈值 / 切句策略 / NLI 模型 / 配对策略 |

## 六、下一步可做的改进

按工作量从小到大排：

1. **过滤无效句子** —— 长度 < 4、以问号结尾、纯感叹词（"嗯"/"啊"/"好吧"）的句子不送 NLI。预计能消掉一半 FP。
2. **双向 NLI 验证** —— 同时检查 NLI(p, h) 和 NLI(h, p)，两个方向都判 contradict 才算命中。预计 FP 大幅下降。
3. **置信度组合规则** —— 不只看 contradict 概率，要求 `contradict ≥ 0.5 且 entail ≤ 0.1 且 neutral ≤ 0.4`。把 NLI 模型不确定的 case 归到中性。
4. **加 Claim Extractor 层** —— 用本地小 LLM (Qwen2.5-0.5B) 把 b2 抽成原子命题再判，去掉句法噪声。
5. **接 Goal/Persona 提取** —— 第一轮 b1 当作隐式 persona，引入"persona 一致性"NLI 检查，开启 class 2 的检测。
6. **训练任务相关的 NLI** —— 用 CDConv 训练集 fine-tune Erlangshen，学会区分"话题切换"与"事实矛盾"。这是论文里最值得做的事。
7. **第二次评测换更大测试集** —— 现在的 200 条测试集太小，方差大。建议至少 1000 条，或者跑全集 11660 条（CPU 上估算 ~20 分钟）。
