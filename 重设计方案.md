# 智能体对话不一致检测 — 重新设计方案

> 本文是对原 `Ls-trash` 项目（基于 prompt + glm-4-flash 的端到端方案）的方法论级重新设计。
> 不在原代码基础上修补，而是从问题建模出发推倒重来。

---

## 一、问题陈述

检测智能体多轮对话中的"不一致"，包含两个语义截然不同的层级：

| 层级 | 含义 | 本质 |
|---|---|---|
| **硬冲突** | 事实错误、角色混淆、历史自相矛盾 | 逻辑推理问题 |
| **软偏移** | 没有明显逻辑错误，但话题逐渐偏离用户初始意图 | 意图追踪问题 |

这两个问题需要**完全不同的方法论**，必须分别处理。

---

## 二、为什么"prompt 一个小模型"这条路走不通

原方案的核心错误不是 prompt 写得不够好，而是**问题建模错了**：

1. **把异构问题挤进一个分类器**
   "硬冲突"和"软偏移"是两个完全不同的任务——前者是逻辑推理（claim 之间是否矛盾），后者是意图追踪（是否还在原目标上）。强行用同一套四分类标签去框，必然两边都做不好。

2. **让 LLM 一步到位**
   让一个模型同时完成"找冲突点 → 定位是哪一句和哪一句冲突 → 判断冲突类型"，这是已知 LLM 容易翻车的长上下文任务。模型的偷懒（全判 0）就是这种过载的产物，再多 few-shot 也只是创可贴。

3. **glm-4-flash 是错的工具**
   这个量级的模型根本不能稳定做语义对比。即便换成 glm-4-plus 或更强模型，单 prompt 端到端的方案也只是"看起来能跑"，不能 debug、不能定位、不能改进。

4. **CDConv 数据集本身就误导**
   它只有 2 轮对话、合成注入的冲突，**根本不包含"软偏移"标签**。在 CDConv 上刷再高的分也没解决任务书要求的核心问题。原作者实际上只在做"硬冲突"那一半，而且那一半的数据集还非常窄。

---

## 三、重新设计：拆成两条独立的 pipeline

将整个系统切成两个**独立模块**，输出两类不同的告警，互不耦合。

### Track A：硬冲突检测 — Claim Extraction + NLI

**核心思路**：把"找矛盾"还原成它本来的形式——自然语言推理（NLI）。这是 NLP 二十年的成熟问题，有专门的模型和数据集，没必要让 LLM 去重新发明。

```
对话流入
   │
   ▼
[Claim Extractor]   ← LLM (强模型，比如 glm-4-plus / DeepSeek)
   │   把每一轮 bot 回复抽成原子命题：
   │   "我是北京人"、"我每周打三次篮球"、"我妈不让养狗"……
   ▼
[Belief KB]
   持续累积的事实库 = 人设(persona) ∪ 历史所有 claims
   │
   ▼
[NLI 判别器]   ← 微调好的 NLI 模型 (xlm-roberta-large-xnli /
   │            erlangshen-roberta / bge-reranker 之类)
   │   对每条新 claim c:
   │     遍历 KB 中每条已有 claim k，跑 NLI(k, c)
   │     如果 label = contradiction → 命中
   ▼
[分类后处理]
   命中的那条 k 来自哪里 → 自动决定是哪种冲突:
     - k ∈ persona                  → 角色混淆 (label 2)
     - k 来自更早轮次的 claim       → 历史矛盾 (label 3)
     - k 来自当前轮次的另一条 claim → 句内矛盾 (label 1)
```

**为什么这套比 prompt 端到端强**：

- **可解释**：永远能给出"是哪句话和哪句话矛盾"的证据对，不再是黑箱。
- **可分别优化**：claim extractor 烂可以单独换；NLI 模型可以独立微调；KB 召回可以加 BM25/embedding 检索。
- **长对话不爆炸**：随对话长度线性增长，不依赖窗口大小。
- **分类是后处理**：四分类不是模型预测的，而是规则推出来的，准确率天然高。
- **NLI 比 LLM 便宜两个数量级**：DeBERTa-v3 / RoBERTa-large 量级的判别模型在 GPU 上一秒钟几百对，还可以本地跑。

**已知风险点**：

- 中文 claim 抽取的稳定性。建议用强模型（glm-4-plus 或 GPT-4 级）+ structured output（强约束 JSON schema）。
- NLI 模型在"对话语境"上的迁移效果。可能需要在 CDConv 上微调一下，而这正是论文里能写的实验贡献。
- 时态/模态问题（"我以前喜欢狗" vs "我现在喜欢猫"不矛盾）。需要在 claim 里显式带 `time` / `modality` 字段。

---

### Track B：软偏移检测 — Goal Tracking，不是相似度

软偏移最常见的错误做法是"对每轮做 embedding，看与初始 query 的余弦相似度"。**这条路是错的**，因为：

- 自然对话本来就会在子话题间合理跳转，相似度下降不一定是漂移。
- 用户的"意图"和"措辞"是两层东西，embedding 抓的是后者。
- 真正的漂移信号是**用户没获得他想要的东西**，不是"话题不像了"。

**正确的做法**：显式建模用户目标，再追踪每轮是否在向目标推进。

```
对话开始 (用户前 1-3 轮)
   │
   ▼
[Goal Extractor]   ← LLM
   产出一个结构化目标:
   {
     "goal": "用户想了解 Python asyncio 的死锁排查方法",
     "key_concepts": ["asyncio", "deadlock", "排查"],
     "subgoals": ["定位死锁", "分析原因", "修复方案"]
   }
   │
   ▼
对话进行中，每轮:
   │
   ├──► [Alignment Judge] LLM 判断当前轮 ∈ {on_goal, related, off_topic}
   │     给出短理由
   │
   ├──► [Concept Anchor] 检查 key_concepts 是否还在最近 N 轮中出现
   │
   └──► [User Redirect Detector] 简单分类器/规则
         检测用户是否说出 "我不是问这个"/"回到刚才"/"算了" 等
         (这是金标准信号，几乎没噪音)
   │
   ▼
[Drift Trigger]
   规则融合:
     - 连续 k 轮 off_topic → 漂移告警
     - 用户出现 redirect 信号 → 立即告警
     - key_concepts 消失超过 m 轮 → 弱告警
```

这套方案的关键是：**用户的纠错信号是金标准**。在真实数据上做研究时，这条规则一条顶过 10 个相似度模型——因为用户不抱怨就说明没漂移（或漂移可接受）。

## 四、数据集策略

CDConv 只够当 Track A 的单元测试，**绝对不能当全部评测**。建议：

| 用途 | 数据集 |
|---|---|
| Track A 训练/评测 | CDConv（句内/角色/历史矛盾）+ 人工合成的中文 NLI 增强数据 |
| Track A 跨域测试 | DialFact / 自己用 GPT-4 在 PersonaChat 上注入冲突 |
| Track B 评测 | MultiWOZ 中文版（任务型对话有显式 slot，可作为目标 ground truth）+ 人工标注的 ShareGPT 子集 |
| Track B 漂移合成 | 拿任何长对话，用强模型在中段插入跑题片段，构造正负例 |

最值得做的工作是**自己标一个小但真实的中文长对话漂移数据集（500–1000 条就够）**——这既是论文的 contribution，又能避开"在合成数据上刷分"的批评。

---

## 五、模型选型建议

**不要再用 glm-4-flash 做核心**。

| 角色 | 推荐 |
|---|---|
| Claim 抽取 / Goal 抽取 / Alignment 判断 | glm-4-plus、DeepSeek-V3、Qwen-Max 这一档；预算允许就用 GPT-4o / Claude |
| NLI 判别 | `IDEA-CCNL/Erlangshen-Roberta-330M-NLI`、`xlm-roberta-large-xnli`、`bge-reranker-large` |
| 中文 embedding（仅作辅助信号） | `BAAI/bge-large-zh-v1.5`、`maidalun1020/bce-embedding-base_v1` |
| 用户 redirect 检测 | 一个 fastText / 小 BERT 二分类器就够，不需要 LLM |

**关键原则**：能用判别式小模型解决的事，不要用生成式大模型。LLM 只放在**必须做语义抽取**的环节——其他步骤都用确定性的、可调试的、便宜的组件。

---

## 六、写论文 / 毕设角度的可证明贡献

如果照这套架构做，论文里能讲的故事是清晰的：

1. **Decomposition 假设**：硬冲突是 NLI 问题，软偏移是 goal-tracking 问题，端到端 LLM 分类是建模错误。
2. **Track A 的消融实验**：claim extractor 强度 / NLI 模型选择 / KB 检索方式 → 对最终 F1 的贡献。
3. **Track B 的对比实验**：纯 embedding 相似度 baseline vs. goal tracking 方案，在自建数据集上的差距。
4. **可解释性**：展示几个 case，端到端 LLM 只能说"有矛盾"，本系统能精确指出"第 4 轮的 X 与第 1 轮的 Y 矛盾"。
5. **CDConv 上对比 SOTA**：证明 decomposition 方法在公开 benchmark 上至少不逊色，最好显著超过原作者那种纯 prompt 方案。

---

## 七、最小可行落地路径

从零开始，建议这个顺序（每一步都能独立验证）：

1. **先把 Track A 跑通**：不用 KB，只做"两个 bot 回复 → NLI → 是否矛盾"。在 CDConv 上拿到一个 baseline。
2. **加上 claim 抽取**：把整句话拆成原子命题再做 NLI，看 F1 涨多少。
3. **引入 KB 和 persona 字段**：做完整的四分类后处理。
4. **Track B 单独立项**：先标 100 条小数据，跑通 goal extractor + alignment judge 这条主线。
5. **两条线合并成一个统一 API**：输入是对话流，输出是 `[hard_conflicts, drift_alerts]`。

**建议从第 2 步开始动手**——纸上谈兵不如先在 CDConv 上跑出一个干净的 NLI baseline，看看到底比原作者的 prompt 方案高多少，再决定后续投入。

## 附：原方案遗留问题清单

1. **API Key 泄露**：所有 6 个 Python 文件硬编码同一个智谱 API Key，已随 git 提交泄露。重做之前应当先去智谱后台吊销并重新生成。
2. **`evaluate_cdconv.py:127`**：API 失败默认返回 label=0，会让不稳定样本伪装成"模型正确预测了 0"，污染评测。
3. **`draftpaper.py:145`**：prompt 要求模型输出 `thinking` 字段，但代码里 `result_json.get("reason", "")` 取的是 `reason`，永远拿不到推理过程。
4. **6 个脚本大量代码重复**：`load_cdconv_data` / `call_api_for_evaluation` / `run_evaluation` 在多个文件里几乎一模一样。重做时应该有一个清晰的项目结构，而不是把每次实验另存为一个新脚本。
